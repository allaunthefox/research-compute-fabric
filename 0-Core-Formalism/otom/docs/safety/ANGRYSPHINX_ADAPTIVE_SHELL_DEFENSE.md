# AngrySphinx — Adaptive Shell Defense v0

## Purpose

AngrySphinx is an adaptive defensive shell architecture. It is based on the Sphinx pattern:

```text
answer correctly → pass
answer incorrectly → consequence
```

In this system, the consequence is not deletion or violence. A wrong answer escalates the solver into a harder constructive problem.

```text
wrong answer → harder shell → higher-dimensional solve obligation
```

The harder problem should be socially constructive whenever possible, such as protein folding, cancer research, biological constraint mapping, or other high-value scientific search spaces.

## Scalar rest state and n(3)D spawn

At rest, AngrySphinx is a 0D unbound scalar state whose value lies in `ℝ`.

```text
s ∈ ℝ, s₀ = 0
```

The scalar's value domain is one-dimensional, but the primitive itself is pointlike: no internal geometry, no shell, no riddle, and no permanent manifold.

Under authenticated attack pressure, the scalar spawns into an `n(3)D` pressure-weighted defense manifold.

```text
0D scalar rest state
  → authenticated attack pressure
  → n(3)D spawned shell manifold
```

`n(3)D` means the spawned manifold carries three coupled pressure weights per active dimension:

```text
n(3)D = n dimensions × 3 pressure weights
```

The three pressure weights are:

```text
1. compute pressure
2. semantic pressure
3. reality-contract pressure
```

This gives the defense three times the pressure weight of a single-axis shell. Attack pressure is not merely accumulated; it is decomposed across compute, meaning, and local-rule burden before being routed into FAMM memory.

## Origin note — Anubis inversion

The core primitive was inspired by Anubis-style anti-scraping / proof-of-work defense.

The reaction was:

```text
that is a horrifying waste of compute power
```

AngrySphinx keeps the useful part — making unwanted access pay a cost — but inverts the waste:

```text
if compute must be spent,
route it toward useful work
```

So the design is not merely:

```text
burn cycles to prove effort
```

It is:

```text
spend effort on a shell-local problem whose residue can be useful
```

This is the Anubis inversion principle:

```text
proof-of-work → proof-of-useful-pressure
```

## Defense / research allocation invariant

AngrySphinx assumes that a secure research system cannot expose a neutral open surface. Most incoming pressure must be absorbed by defense before any remainder can become research.

The design target is:

```text
60–70% defense / 30–40% research
```

Interpretation:

```text
defense share  = shelling, filtering, gatekeeping, routing, logging, FAMM memory
research share = constructive residue after alignment filtering
```

This is the fence invariant:

```text
no direct hole in the fence
```

Every interaction must either:

```text
1. fail closed
2. route away
3. pass through an alignment gate
4. become useful constructive pressure
```

The reason for the high defense allocation is that a low-defense research interface creates an extraction path. A pure-defense interface wastes compute. AngrySphinx tries to occupy the middle: mostly fence, partially research engine.

## Rooted-system boundary

AngrySphinx is an uncomfortable compromise because it is only useful while the shell boundary still exists.

```text
you cannot use AngrySphinx if the system is already rooted
```

If the attacker has root-equivalent control, the shell can be bypassed, altered, observed, or replayed. In that state, AngrySphinx must not be treated as an active defense.

Rooted-state behavior:

```text
root detected
  → stop treating shell outputs as trustworthy
  → preserve forensic logs if possible
  → fail closed
  → rotate secrets / rebuild from clean substrate
  → do not route rooted outputs into canonical memory or FAMM basins
```

This is the hard threat-model boundary:

```text
AngrySphinx is a fence, not a cure for a captured fencepost.
```

## Counter-rotation gear-reduced shells

AngrySphinx shells are counter-rotating and gear-reduced.

Interpretation:

```text
outer shell rotates against attack pressure
inner shell counter-rotates against direct extraction
gear reduction converts high-speed attack energy into slower, higher-torque solve pressure
```

The shell should not transmit attacker force directly into the protected core. It should reduce, phase-shift, and redirect that force into bounded work.

```text
raw attack velocity
  → counter-rotation
  → gear reduction
  → bounded solve torque
  → constructive residue or fail-closed route
```

This matters because a direct shell becomes a hole when force couples cleanly to the core. A gear-reduced shell forces the attacker to pay mismatch losses and solve intermediate constraints before reaching anything canonical.

## The FAMM twist

FAMM means Frustration Aligned Memory Management. In normal use, FAMM is helpful: it records route pain so future routing gets better instead of repeating bad paths.

In AngrySphinx, that helpful property becomes a sharp defensive twist.

A bad interaction does not merely fail. It becomes route memory.

```text
bad route
  → normalized artifact
  → fingerprint
  → route outcome
  → FAMM basin / hold / scar
  → future shell selection changes
```

The more an adversary pushes, the more information it gives the defense about which routes hurt, which routes fail, which routes create torsion, and which shells should harden.

This is nasty in context because the attacker is training the fence against itself.

```text
attack attempt
  → solve pressure
  → route-memory scar
  → stronger future shell
```

FAMM must never ingest unnormalized or rooted outputs. The artifact-normalization rule still applies:

```text
artifact → fingerprint → route → outcome → memory
```

Unnormalized artifacts may be archived, but cannot create basins, scars, or routing priors. Rooted outputs cannot update canonical memory or FAMM basins.

## Spite as design pressure

AngrySphinx was designed by asking:

```text
what would be the most spiteful thing I could ever see?
```

This is not a license for retaliation. In the spec, spite is treated as a design-pressure heuristic:

```text
maximize adversary regret
without damage, coercion, retaliation, or corruption of canonical state
```

The safe translation is:

```text
misaligned effort should feel unrewarding,
expensive,
embarrassing to justify,
and likely to improve the defender's route memory.
```

A successful spite-pressure design makes the adversary conclude:

```text
this was not worth it
```

while the system concludes:

```text
route memory improved
research residue may be useful
canonical state stayed clean
```

## Alignment refusal principle

AngrySphinx encodes a distaste for misaligned work.

```text
misaligned work does not receive a neutral interaction surface
```

If an actor or process insists on imposing cost, distraction, extraction, or hostile work on the system, it receives one of two outcomes:

```text
1. disengage / go away
2. be routed into constructive solve pressure
```

The constructive solve pressure should target humanity-positive problem classes whenever possible.

```text
if you are going to consume attention or compute,
you either stop,
or you help solve real problems
```

This is not retaliation. It is alignment filtering and cost redirection.

## Core thesis

AngrySphinx converts adversarial or low-quality solving into useful burden.

```text
attack pressure
  → shell question
  → answer check
  → if wrong: route to harder constructive solve
  → record route cost in FAMM
```

## Mechanism

A shell has:

```text
question
answer_check
escalation_target
constructive_problem_class
exit_condition
route_memory
```

Wrong answers do not merely fail. They increase the required solve class.

## Escalation ladder

| Attacker / solver class | Shell response |
|---|---|
| toy / Atari-class | basic adder or arithmetic shell |
| ordinary automation | constrained symbolic shell |
| high compute / supercomputer | law-definition or reality-contract shell |
| universe-scale / maximal adversary | constructed universe-shell with alien physics rules |

## n(3) division

`n(3)` is the three-axis escalation pressure:

```text
n(3) = compute burden + semantic burden + reality-contract burden
```

Each wrong answer increases at least one division.

```text
n3_next = n3_current + Δcompute + Δsemantic + Δreality
```

In the spawned manifold, this becomes `n(3)D`: every active dimension is pressure-weighted by the three-axis division. This is what makes the shell three times heavier than a single-pressure model.

The design goal is:

bad solve attempts become useful search pressure

## Audit-log feedback layer

AngrySphinx assumes that everyone keeps logs.

Therefore the log file becomes part of the interaction surface. It is a corrective feedback layer: a record of bad behavior, bad routing, or hostile extraction attempts that makes the interaction unpleasant to revisit, classify, or justify.

This is the "whack on the nose" mechanic:

```text
act like an ass
  → get logged inside an irritating / embarrassing / high-friction shell
  → future inspection pays the social/cognitive cost of that behavior
```

The log layer is not the primary defense and not cryptography. It is behavioral feedback through unavoidable audit trail.

## Cringe-load moat

AngrySphinx may include aversive semantic padding: deliberately embarrassing, self-insert, cringe, or socially costly log material.

This is not cryptography. It is operator friction.

The rule is:

```text
cringe may raise extraction cost, but must not corrupt canonical signal
```

Therefore logs must distinguish:

```text
canonical_signal
decoy_log
fiction_shell
aversion_padding
route_memory
behavioral_feedback_log
```

## Safety boundary

AngrySphinx is not a substitute for:

- encryption
- access control
- backups
- sandboxing
- signatures
- formal verification
- ordinary security hardening

It is a defensive cost-shaping and constructive redirection layer.

## Misalignment boundary

AngrySphinx must not become harassment, coercion, retaliation, or malware.

Allowed:

- refusal
- isolation
- rate limiting
- puzzle gates
- benign tarpits
- synthetic/logical shells
- constructive research-task redirection
- FAMM route-memory recording
- behavioral feedback through local audit logs

Disallowed:

- damaging systems
- targeting people
- extortion
- doxxing
- unauthorized persistence
- retaliatory payloads
- making medical claims from unvalidated outputs
