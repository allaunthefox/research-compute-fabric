# Translator Packets v0.1

## Status

**Claim state:** `BEAUTIFUL_PROVISIONAL`

This spec documents the method that emerged while translating metaphor-heavy cognitive material into OTOM mathematical operators.

A **Translator Packet** is a compressed semantic packet that contains enough latent structure to be unfolded into:

```text
image / phrase / metaphor
→ operator motif
→ mathematical type
→ trigger predicate
→ gate
→ equation or update rule
→ Warden boundary
→ Lean theorem shape
```

The goal is not to treat metaphor as proof. The goal is to preserve useful structure while preventing overclaim.

## Why this matters

Some metaphor-heavy inputs are not random noise. They can carry hidden operator structure:

```text
object
transformation
trigger
failure mode
receipt
boundary
```

Examples from the recent OTOM build:

| Surface packet | Operator extracted | Formal role |
|---|---|---|
| `turned round and round and upside down` | Torsion Flip Operator | rotation accumulates torsion, crosses threshold, flips orientation |
| Connect Four pieces slide off + `SORRY` | SORRY Collapse Gate | support predicate failure causes lattice evacuation and invalidation receipt |
| Delphi / Delta Phi / IDPC | Oracle Interrogation IDPC | fallible oracle exposes pattern candidates under phase displacement |
| standing-wave contour rings | Kinetic-Sidon standing wave lattice | wave motion becomes contour events and Sidon pair addresses |
| creepy smiles / navigation oracle | False-Affect Oracle Gate | interface reassurance is checked against invariant support |

The recurring pattern is:

```text
metaphor is not the result
metaphor is the compressed carrier
translation extracts the operator
receipts decide promotion
```

## Translator packet anatomy

A useful packet usually contains at least four of the following:

```text
1. Object carrier
   What thing is being acted on?

2. Transform verb
   What happens to the object?

3. Trigger condition
   When does the transform occur?

4. Failure or inversion mode
   What breaks, flips, collapses, aliases, or resets?

5. Receipt marker
   What visible/signaled marker records the transition?

6. Boundary condition
   What must not be overclaimed?

7. Reconstruction path
   How could the transformed object be decoded, replayed, or checked?
```

A packet is stronger when it contains object + transform + trigger + receipt.

A packet is weaker when it only contains mood or association.

## Translation pipeline

### Step 1 — Preserve the surface

Record the raw packet without correcting it too early:

```text
raw phrase / image / remembered scene / accidental word chain
```

This prevents premature normalization from deleting the signal.

### Step 2 — Extract operator motif

Ask:

```text
What operation is being implied?
```

Examples:

```text
round and round      → repeated rotation / winding
upside down          → inversion / orientation flip
pieces slide off     → support collapse / evacuation
SORRY                → invalidation receipt
creepy smile         → affect/invariant mismatch
```

### Step 3 — Assign mathematical type

Choose the type of object being transformed:

```text
scalar field
vector
local frame
lattice state
pair signature
oracle response
receipt ledger
support predicate
```

This is the main translation step. Many bad translations fail because they skip type assignment.

### Step 4 — Define trigger predicate

Every operator needs a condition:

```text
torsionScore ≥ torsionThreshold
supportStatus = invalid
aliasCount = 0
receiptExists = true
confidence > evidenceSupport
```

A trigger converts metaphor into a gate.

### Step 5 — Define allowed transition

Write the update rule:

```text
if trigger holds:
  transformed state
else:
  preserved state
```

Examples:

```text
if torsion crosses threshold: orientation := inverted
if support fails: tokens := [] and receipt := SORRY
if proof receipt absent: status cannot become REVIEWED
```

### Step 6 — Add Warden boundary

Separate intuition from claim:

```text
Allowed: the metaphor motivated an operator name and placement.
Blocked: the metaphor proves a law.
```

This is mandatory. Without a Warden boundary, Translator Packets become story inflation.

### Step 7 — Produce theorem shape

Translate the gate into a Lean-compatible theorem:

```text
invalid_support_emits_sorry
apply_flip_inverts_when_threshold_met
sidon_like_no_alias_collision
literal_failure_can_still_be_pattern_candidate
```

The theorem should prove the finite gate, not the whole metaphor.

## Core packet schema

A Translator Packet can be recorded as:

```text
Packet:
  rawSurface: original phrase/image/memory
  objectCarrier: thing being transformed
  transformVerb: action implied by the packet
  triggerPredicate: condition that activates the transform
  outputState: result of the transform
  receiptMarker: observable record of transition
  forbiddenClaim: what must not be inferred
  allowedClaim: bounded interpretation
  theoremShape: finite proposition worth formalizing
```

## Translation load estimate

Recent OTOM examples suggest the translation is often not arbitrary invention. Many packets already contain strong latent structure.

Approximate breakdown:

```text
60% latent structure already present in metaphor packet
25% formal language translation / type assignment
15% Warden boundary and proof-gate construction
```

For equation-heavy cases like the two-layer kinetic/Sidon lattice:

```text
40% latent structure already present
45% formal/equation construction
15% boundary control
```

The harder the case, the more work happens in type assignment and gate construction.

## Packet quality levels

### Level 0 — Noise

```text
No stable object, transform, trigger, or receipt.
```

Action:

```text
Do not formalize.
```

### Level 1 — Motif

```text
A recognizable image or phrase, but no clear trigger.
```

Action:

```text
Record as intuition only.
```

### Level 2 — Operator candidate

```text
Object + transform are clear.
```

Action:

```text
Write a bounded spec.
```

### Level 3 — Gate candidate

```text
Object + transform + trigger + output are clear.
```

Action:

```text
Add Lean scaffold for finite gate.
```

### Level 4 — Receiptable artifact

```text
Gate is formalized, tested, and connected to receipts or measurements.
```

Action:

```text
Candidate for CALIBRATED_ENGINEERING_DELTA.
```

### Level 5 — Reviewed claim

```text
Formal proof / benchmark / measurement receipt supports the claim.
```

Action:

```text
Candidate for REVIEWED.
```

## Relation to Oracle IDPC

Translator Packets are a practical version of the fallible-oracle doctrine.

The oracle is not trusted because it is right. It is useful because a partial, wrong, strange, or accidental surface can expose a deeper pattern candidate.

```text
mostly wrong surface
→ sometimes right enough
→ operator candidate
→ gate
→ receipt decides promotion
```

This is exactly why the metaphor must be preserved at first, but not believed without formal gates.

## Warden boundaries

### Blocked claims

```text
Metaphor is proof.
Randomness is truth.
A remembered phrase establishes a physical law.
Every evocative image deserves formalization.
A Lean scaffold proves the original metaphor was objectively intended.
```

### Allowed claims

```text
Metaphor can be a compressed carrier of operator structure.
Random surfaces can be inspected as fallible oracle outputs.
A packet may motivate a bounded toy formalism.
Formal gates can preserve useful structure while blocking overclaim.
Lean can check the finite predicate/update/theorem shape extracted from the packet.
```

## Strongest formulation

> A Translator Packet is a compressed cognitive object that becomes mathematically useful only after it is decompressed into typed state, trigger predicates, bounded transforms, Warden constraints, and receipt-bearing theorem shapes.
